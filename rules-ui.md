# Правила разработки UI

## Ограничения UI

- UI не придумывает данные: он только показывает и отправляет их.
- Читать и писать данные можно только через топики binder.
- Файлы `api/*` изменять нельзя.
- Если данных нет, показывается пустое состояние, без выдумывания данных.

## Структура проекта

- `index.html` - это каркас: основные контейнеры страницы.
- Содержимое вкладок и панелей описывается в `index.html` или создаётся в JS-файлах - по согласованию.
- Каждый крупный компонент UI = собственный JS-модуль.
- Для каждого комопнента должны быть 2 CSS-файла:
- `<component>-layout.css` отвечает за расположение элементов.
- `<component>.css` отвечает за цвет, фон, отступы, шрифты и эффекты.
- Все CSS подключаются только через `css/layout.css`.
- Добавил блок -> добавь JS, CSS и подключи CSS в `css/layout.css`.
- Поменял структуру -> обнови `struct.md` и `ui-struct.md`.

## Компоненты и состояние

- Компонент должен быть классом с `constructor`, `initialize` и `element`.
- `constructor` создает DOM-элементы и внутренние поля.
- `initialize` вешает события, подписки на топики и применяет данные.
- `element` возвращает корневой DOM-узел компонента.
- Селекторы, topic names и лимиты должны быть в `const` в начале файла.
- Для связи HTML и JS используются `data-*` атрибуты.
- Текст в DOM пишеться через `textContent`, а не через `innerHTML`.
- Перед работой с элементом должна быть обработка `null` и `undefined`.
- Значения из топиков приводиться к нужному типу: `boolean`, `number`, `string`.
- После подписки сразу применяется текущее `topic.value`.
- Для частых событий (`input`, `slider`) устанавливется частота отправки.
- Тяжелые элементы (например, 3D-модель) инициализируется один раз и переиспользуется.
- Локально храниться только UI-состояние: вкладка, режим, вид модели.

## UX: эвристики Нильсена

- Видимость статуса: пользователь всегда понимает, что происходит сейчас.
- Показывается: загрузка, успешно, ошибка, нет данных.
- Соответствие реальности: используются понятные слова и единицы измерения.
- Контроль у пользователя: есть отмена, пауза, стоп и безопасный выход.
- Консистентность: одинаковые действия выглядят и работают одинаково.
- Предотвращение ошибок: блокируются невозможные и опасные действия.
- Узнавание, а не память: подписи, подсказки и значения по умолчанию.
- Эффективность: частые действия выполняются с минимумом кликов.
- Минимализм: убираются лишние элементы и декоративный шум.
- Помощь при ошибках: короткий текст и шаг, что делать дальше.
- Справка в интерфейсе: локальные подсказки рядом со сложной функцией.

## Верстка и доступность

- Цвета, размеры и отступы берутся из `css/vars.css` для единого стиля.
- Не добавлять случайные "магические" значения без причины.
- Раскладка строиться на grid-layout, чтобы блоки вели себя предсказуемо.
- В flex/grid-контейнерах ставиться `min-height: 0`, если внутри нужен скролл.
- UI-состояния задаёться классами: `is-active`, `is-hidden`, `is-open`, `is-bad`.
- У каждой кнопки ставитья `type="button"`, чтобы не было случайного submit.
- Для доступности используется ARIA: `aria-label`, `aria-pressed`, `aria-expanded`.
- Для каждого блока должны быть состояния: normal, loading, empty, error.
