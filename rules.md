# Правила разработки и репозиториев

## Репозитории и ветвление

- Все проекты должны быть размещены в GitLab.
- Для каждого проекта должен быть отдельный каталог проекта, в котором находятся репозитории.
- В каждом репозитории:
    - ветка по умолчанию — `develop`;
    - весь код должен находиться в репозитории.
- Ветки:
    - `main` — релиз;
    - `develop` — отладка на изделии/железе;
    - для реализации фич — отдельные ветки.

## Коммиты, пуши, теги

- Коммиты выполняются всегда, даже если код еще не отлажен.
- Коммиты делаются не только по завершению фичи, но и в процессе разработки.
- При уходе с рабочего места более чем на час должны быть сделаны все коммиты и пуши.
- Pull/Push выполняются ежедневно.
- После реализации фичи выполняется merge в `develop`.
- После успешной отладки выполняется merge из `develop` в `main`.
- Для каждой версии пакета создается новый тег с версией.

## Документация в репозитории

- В каждом репозитории должен быть комплект файлов:
    - `struct.md` с описанием структуры файлов и кратким описанием каталогов и файлов;
    - `readme.md` с описанием проекта, целей, принципов работы, задач, используемых методов, фреймворков, стека и других аспектов.

## IDE и конфигурация

- Основная IDE — VS Code.
- Исключения: проекты, где VS Code не позволяет выполнить полный цикл отладки (например, прошивки) — по согласованию.
- В проекте обязательно должна быть папка с единой конфигурацией VS Code, используемой в команде.

## Общие требования к коду

- Парадигма — ООП, код пишется с использованием классов.
- Исключения: точки входа и языки, не поддерживающие классы (например, C).
- Методы и функции — не более 30 строк кода.
- Длина строки кода — не более 100 символов.
- Имена переменных, классов, методов, модулей и т.п. должны полностью раскрывать смысл и состоять из несокращенных слов (кроме общепринятых аббревиатур).
- Каждый класс должен быть в отдельном файле.
- Исключение: зависимые классы, которые нельзя унифицировать.
- Члены класса, не предназначенные для внешнего использования, должны быть приватными:
    - JS: префикс `#`;
    - Python: префикс `__`.

## Стиль именования

- Общие правила:
    - Переменные короче 4 символов запрещены, если это не общепринятые аббревиатуры или полные слова.
    - Имена должны быть без ошибок.
    - Имена файлов `js`, `json`, `md` и каталогов должны быть в стиле `kebab-case`.

### Именование (JS)

- Методы: `camelCase` (первая строчная).
- Классы: `CamelCase` (первая заглавная).
- Свойства/поля/переменные: `camelCase` (первая строчная).
- Константы: `SNAKE_CASE` (в верхнем регистре).
- Приватные методы должны начинаться с `#`.

### По языкам и форматам

- JS, JSON: `lowerCamelCase`
- Python, YAML: `snake_case`
- C, C++, C#: `CamelCase`

## Языковые требования

### JavaScript

- Используем только ECMA Script 2022 и выше.
- Версия Node.js — v24.
- Для unit-тестов используем встроенный test runner Node.js 20+.
- Все публичные методы и свойства должны иметь 100% аннотацию типов (JSDoc).

### Python

- Версия 3.10 и выше.

### C#

- Рантайм — .NET (Core) 8.0 и выше.

## Процесс разработки

- Структура и прототипы объектов.
- Согласовывание с участниками группы.
- Реализация тестов/отладка через тесты.

## Конфигурация и обмен данными

- Конфигурационные файлы должны быть в формате JSON.
- При обмене данными на прикладном уровне должен использоваться формат JSON.

## Ограничения и запреты

- Запрещено использовать технологии GTK, GDI и другие на их основе.
- Запрещено использовать сторонние модули, если есть встроенный функционал платформы/языка/фреймворка.
- Запрещено создавать потоки, если задачу можно решить без потоков или нет задачи оптимизации вычислений.

## Требования к развертыванию

- Любой прикладной проект должен быть адаптирован под работу в Docker-контейнере.
- Любой проект должен быть адаптирован к запуску в неограниченном количестве инстансов.
- Конфигурирование каждого инстанса выполняется через переменные окружения, передаваемые через environment в `docker-compose` YAML.

## Модели (3D/САПР)

- Все модели, используемые в проекте, должны быть в репозитории.
- Минимальный комплект форматов на модель:
- `step` — исходник CAD (если есть).
- `ipt` — исходник Inventor (если есть).
- `blend` — исходник Blender, подготовленный для экспорта.
- `glb` — финальный экспортный формат для ПО.
- Имена файлов моделей должны совпадать между форматами, различается только расширение.
- Для каждой модели должен быть единый источник правки (`step`/`ipt`/`blend`), остальные форматы — экспорт.
- Экспорт в `glb` выполняется из актуального исходника.
- В репозитории должны храниться только актуальные версии моделей.
